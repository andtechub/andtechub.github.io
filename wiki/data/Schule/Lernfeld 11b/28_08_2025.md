## Lösungen / Ausarbeitungen

### Lernfeld 10b – Unterschiede Server-Hardware vs. End-User-Hardware

| Komponente | Server-Hardware | End-User-Hardware | Wesentlicher Unterschied / Nutzen |
|------------|-----------------|-------------------|------------------------------------|
| Prozessor (CPU) | Multi-Socket fähig, hohe Kernanzahl, größere Caches, Unterstützung für AVX-512, häufig Intel Xeon / AMD EPYC | Einzel-Socket, weniger Kerne, kleinere Caches, Core i5/i7/Ryzen | Skalierbarkeit, Parallelisierung, Stabilität, längere Produktverfügbarkeit |
| Zuverlässigkeit / Features | ECC-Unterstützung, RAS-Features (Machine Check Architecture, Memory Scrubbing) | Kein ECC (Ausnahme: einige Ryzen + ECC-taugliche Boards) | Reduktion von Silent Data Corruption |
| Arbeitsspeicher | ECC, Registered (RDIMM), Load-Reduced (LRDIMM), hohe Kapazitäten (Terabyte-Bereich), Hot-Plug möglich | Unbuffered DIMMs (UDIMM), geringere max. Kapazität, kein Hot-Plug | Stabilität bei 24/7-Betrieb, Kapazität, Fehlertoleranz |
| Massenspeicher | Enterprise SSDs (NVMe U.2 / U.3, PCIe Gen4), SAS-HDDs 10k/15k RPM, RAID-Controller mit Cache & BBU, hohe DWPD | SATA-SSDs (Consumer TLC/QLC), Desktop-HDD 5.4k/7.2k RPM, Onboard-Controller | Haltbarkeit (Endurance), IOPS, Fehlersicherheit, konsistente Latenzen |
| Storage-Schnittstellen | SAS 12/24 Gbit/s, NVMe Backplanes, Hot-Swap Schächte | SATA Onboard, wenige M.2 Slots | Wartbarkeit & Austausch im Betrieb |
| Netzwerkkarten (NIC) | Dual-/Quad-Port 1/2.5/10/25/40 Gbit, SR-IOV, Offloading (TCP/UDP checksum, VXLAN), Remote Management (iLO/iDRAC) | 1× 1 Gbit Onboard, selten 2.5/10 Gbit | Bandbreite, Virtualisierung, Verwaltung |
| Mainboard / Chipsatz | Server-Chipsatz, IPMI / iDRAC / iLO, out-of-band Management, redundante Firmware, längerer Lifecycle | Consumer-Chipsatz, kein dediziertes Management | Fernwartung & Stabilität |
| Stromversorgung | Redundante PSU (Hot-Plug), höhere Effizienz (80Plus Platinum/Titanium) | Einzelnes Netzteil, Bronze/Gold | Verfügbarkeit (N+1), Energieeffizienz |
| Gehäuse / Formfaktor | Rack (1U–4U) oder Tower-Server, optimierter Luftstrom, Hot-Swap Lüfter | ATX/Mid-Tower, manuelle Wartung | Dichte, Wartbarkeit |
| Kühlung | Luftkanäle, geregelte Hochleistungslüfter, teilweise Liquid Cooling in HPC | Standard-Lüfter | Dauerlasttauglichkeit |
| Virtualisierung | Zertifizierte Hardware (VMware, Hyper-V, Proxmox), IOMMU, SR-IOV | Teilweise vorhanden | Garantierte Kompatibilität |
| Sicherheit | Secure Boot, TPM 2.0, Firmware Signing, Chassis Intrusion Detection | Teilweise Secure Boot, TPM optional | Manipulationsschutz |
| Lebenszyklus | 3–7 Jahre Verfügbarkeit/Ersatzteile, definierte Roadmaps | Schnell wechselnde Generationen | Planbarkeit, Support |

### Angebot – Ersatz eines bestehenden KMU-Servers

Hinweis: Die konkrete Anforderungsseite lag nicht vor. Annahmen wurden getroffen, basierend auf typischem KMU (ca. 25–40 Benutzer, Dienste: Active Directory, Fileserver, Print, optional kleinere Virtualisierung / Applikationsserver, zukünftige Erweiterbarkeit). Backup, Support & Wartung laut Aufgabenstellung noch nicht berücksichtigt.

#### Annahmen
- Ziel: Konsolidierung bisheriger Dienste auf einem virtualisierungsfähigen Host (2–4 VMs geplant)
- Verfügbarkeit: 8×5 + hohe interne Verfügbarkeit (kein Full-HA-Cluster gefordert)
- Wachstum: +30 % Speicherbedarf in 3 Jahren
- Performance: Fokus I/O für File & AD + moderate Applikationen
- Sicherheitsanforderungen: Verschlüsselung möglich (BitLocker / Software), Remote Management, ECC-RAM

#### Begründung der Architektur
- Trennung von System-/Boot-Storage und Daten-Storage erhöht Wartbarkeit
- RAID1 für System → schnelle Wiederherstellung / geringes Risiko
- RAID10 für Daten → Performance + Ausfallsicherheit
- 10 Gbit NIC für zukünftige Skalierung (z.B. iSCSI / Backup / höheres Fileshare-Volumen)
- 64 GB RAM ausreichend für Start; Clusterfähigkeit später durch zweiten Host möglich
- Hyper-V genügt (keine zusätzlichen Lizenzkosten, geringe Komplexität)

#### Erweiterungspotenzial
- Zusatz-CPU → Verdopplung Kerne
- RAM-Ausbau auf 128 / 256 / 512 GB
- Zusätzliche NVMe für Tiering / Cache
- Zweiter Host + Shared Storage → HA / Live Migration

#### Nicht enthalten (kann nachgereicht werden)
- Backup-Konzept (Software wie Veeam / Windows Server Backup / Proxmox Backup)
- Patch- und Härtungskonzept (Baseline CIS / BSI)
- Wartungsvertrag / SLA
- Security (EDR, zentrale Protokollierung, SIEM-Anbindung)

### Lernfeld 11 – Vorgehen Strukturanalyse (Kurzleitfaden)

1. Physische Geräte erfassen (Typ, Hersteller, Standort, Funktion)
2. Logische Zuordnung (IP, VLAN, Subnetze, Dienste)
3. Abhängigkeiten (Server → USV → Switch → Router)
4. Schutzbedarf je Komponente (Verfügbarkeit / Vertraulichkeit / Integrität)
5. Dokumentation in Vorlage (Versionieren!)

Empfohlene Kategorien in der Vorlage:
- Bezeichnung / Inventarnummer
- Funktion / Rolle
- Netz (IP / VLAN / Segment)
- Dienste / Ports
- Schutzbedarf (niedrig / mittel / hoch) mit Begründung
- Verantwortlicher
- Bemerkungen (EOL, Firmwarestand)
